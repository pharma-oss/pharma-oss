# pharma-oss 運用における課題・問題点

本システム（pharma-oss）は、Local-Firstアーキテクチャ、ブラウザ完結型の処理（OCRやレセプト生成）を特徴としていますが、実際の薬局業務で運用するにあたって以下の課題やリスクが想定されます。

## 1. データ永続性とローカルストレージ（IndexedDB）のリスク
- **データ消失リスク:**
  本システムはRxDBとDexieを用いてブラウザのIndexedDBにデータを保存しています。OSの空き容量不足時や、ユーザーが誤ってブラウザの履歴・キャッシュをクリアした場合、患者データや薬歴などの重要データが消失する恐れがあります。
- **バックアップと永続化の必要性:**
  ローカルデータのみに依存するのは危険なため、クラウドや店舗内サーバーへの確実な自動バックアップ（同期）と、Storage APIの `navigator.storage.persist()` などによるブラウザのデータ自動削除を防止する対策が必須となります。

**✅ 解決済み:**
- `DatabaseProvider` の初期化時に `navigator.storage.persist()` を呼び出すことで、ブラウザ側のストレージ圧迫時の自動削除を防止し、オフラインデータの永続性を確保しました。
- DB初期化失敗時にIndexedDBを自動削除する旧「自己修復」処理を無効化しました。現在は復旧通知を保存し、画面に手動復旧が必要な状態を表示します。

## 2. セキュリティとプライバシー（機微情報の保護）
- **ローカル保存データの非暗号化:**
  IndexedDBに保存されるデータ（患者の個人情報や処方内容などの要配慮個人情報）は、標準では暗号化されていません。端末の盗難やマルウェア感染時、あるいは共用PCにおいて、他のユーザーや開発者ツールから容易に閲覧・抽出される危険性があります。RxDBの暗号化プラグインなどの導入検討が必要です。

**✅ 解決済み:**
- `rxdb/plugins/encryption-crypto-js` を用いて、IndexedDBに保存される要配慮情報の暗号化を実装しました。
- `crypto.randomUUID()` 等による推測不可能なUUIDを主キーに採用し、IDOR（Insecure Direct Object Reference）脆弱性を防止しています。
- OCR時のファイルアップロードにおいて、MIMEタイプの検証とファイルサイズ制限（10MB）を実施し、不正ファイルによるXSSやDoS攻撃を防止しています。
- スタッフが未登録の初回のみ初回セットアップモードでログインを回避し、スタッフ登録後は通常ログインを必須化しました。操作者情報はセッション保存にし、ブラウザ再起動後は再ログインを求めます。
- パスワードハッシュをPBKDF2-SHA-256へ移行し、既存の旧SHA-256ハッシュはログイン検証時のみ互換対応します。モックパスキーと弱い乱数フォールバックは開発環境限定にしました。

## 3. クライアント端末のパフォーマンス負荷
- **ブラウザでの高負荷処理（OCRなど）:**
  Tesseract.jsを用いたブラウザ内での処方箋OCR処理や、UKE（電子レセプト）ファイルの生成処理は、端末のCPUやメモリに大きく依存します。薬局に設置されている一般的な（低スペックな）事務用PCでは、処理遅延やブラウザのフリーズ、メモリ不足によるクラッシュが発生する懸念があります。
- **OCR言語データのダウンロード:**
  Tesseractの日本語モデル等の学習データ読み込みにより、初期起動時やキャッシュクリア後のネットワーク帯域消費とロード時間が増大します。

**✅ 解決済み:**
- Tesseractモデルとスケジューラーの初期化をバックグラウンドで事前実行する仕組みを導入し、コールドスタート時のレイテンシを解消しました。
- 重い依存関係の動的インポート（Dynamic Imports）を活用し、初期JavaScriptバンドルサイズの削減とTime to Interactive(TTI)の最適化を行いました。

## 4. 文字コードと外部連携規格の問題
- **UKEファイルの文字コード（Shift-JIS要件）:**
  コードベース（`src/lib/receipt/uke_generator.ts`）内でも言及されていますが、現在のUKEファイル生成処理はUTF-8で行われています。しかし、日本の標準的な医療保険請求（レセプト電算処理システム）では **Shift-JIS** でのファイル出力が求められます。運用開始前に、ブラウザ側でShift-JISへエンコード出力する対応（外部ライブラリの導入など）が不可欠です。

**✅ 解決済み:**
- `encoding-japanese` ライブラリを利用し、UKEファイル生成時に UTF-8 から Shift-JIS へのエンコード出力を行うように対応しました。

## 5. データベースのバージョン管理とマイグレーション
- **クライアントサイドでのスキーマ移行リスク:**
  機能追加などに伴いDBスキーマ（`src/db/schema.ts`）を変更する場合、バックエンドのDBを更新する一般的なWebアプリとは異なり、各クライアント端末のローカルDB上で個別にマイグレーション処理を実行する必要があります。特定の端末でマイグレーションが失敗した場合のリカバリ手順や、新旧バージョンが混在した場合のシステム全体の不整合を防ぐ運用フローの構築が求められます。

**✅ 解決済み:**
- `rxdb/plugins/migration-schema` を導入し、クライアントサイドでのデータベースバージョンアップに伴うマイグレーションロジックを実装しました。これによりスキーマ変更時の不整合を防止しています。

## 6. 分散環境でのデータ競合とネットワーク制約
- **コンフリクト（競合）解決の難しさ:**
  オフライン動作やP2P同期をサポートする設計上、ネットワーク切断時に複数の端末から同じ患者の薬歴やステータスを同時に更新した場合、再接続時にデータの競合が発生します。業務上矛盾が生じない競合解決ルールの明確化が必要です。
- **P2P通信のファイアウォール問題:**
  店舗内でのデータ同期にWebRTCなどのP2P通信を利用する場合、薬局内の強固なファイアウォールやセキュリティソフトによって通信が遮断されるリスクがあります。
