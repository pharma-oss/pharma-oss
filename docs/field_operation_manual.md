# pharma-oss 現場運用マニュアル

このマニュアルは、店舗で毎日システムを操作するスタッフ（薬剤師・事務）向けの手順書です。
「いつ・誰が・どの画面で・何をするか」を業務の時間割に沿ってまとめています。
機能ごとの詳しい説明は [取扱説明書（user_manual.md）](user_manual.md)、開発・保守は [developer_manual.md](developer_manual.md) を参照してください。

> **実請求前の注意**: 電子レセプト（UKE）出力は、公式仕様との突合と返戻ケースの検証が完了するまで実請求に使用しないでください（設定 > レセプト点検 で未検証項目を確認できます）。

## 1. 役割と権限の目安

| 業務 | 事務 | 薬剤師 | 管理者 |
|---|---|---|---|
| 受付（OCR/手入力/電子処方箋） | ○ | ○ | ○ |
| ピッキング・GS1照合 | ○ | ○ | ○ |
| 処方監査・薬歴（SOAP）・薬剤師確認 | × | ○ | ○ |
| 算定変更・UKE出力 | × | ○ | ○ |
| バックアップ書き出し/復旧 | × | ○ | ○ |
| スタッフ管理・施設基準・マスタ更新 | × | △（設定による） | ○ |

権限の割り当ては「設定 > 初回セットアップ > 権限ロール設定」で店舗の運用に合わせて調整できます。

## 2. 開局前チェック（毎朝 5分）

1. **端末とログイン**: 各端末でアプリを開き、自分のアカウントでログインします。共有アカウントは使いません。
2. **ダッシュボード確認**: 「要対応」表示（バックアップ未完了・返戻・期限超過フォロー）が残っていないか確認します。
3. **前日のバックアップ**: ダッシュボードまたは「設定 > バックアップ」で、前日の暗号化バックアップと外部保存確認が記録済みであることを確認します。未完了なら開局前に実施します。
4. **復旧通知の確認**: 画面にデータベース復旧が必要な通知が出ている場合は操作を止め、管理者がバックアップからの復旧手順（user_manual §3.8）を実施します。
5. **マスタ更新の予定確認**: 薬価改定や医薬品マスター更新の予定日であれば、開局前に「設定 > マスタ更新」で取り込み、差分CSVとロールバックJSONを保管します。

## 3. 受付〜完了の基本フロー（1患者ごと）

### 3.1 受付
- **紙処方箋**: 「処方箋OCR」で読み取り、患者・処方元・薬品・用法を目視確認して確定します。OCR信頼度が低い項目は必ず原本と照合します。
- **手入力**: OCRなし受付では氏名と生年月日で既存患者へ名寄せされます。保険番号だけで判断しません。
- **電子処方箋**: 受付入力の「電子処方箋」欄から取得します。「処方内容（控え）」だけを見て調剤しないでください（詳細は user_manual §3.3）。
- 受付が終わると、患者はダッシュボードの対応キューに載ります。

### 3.2 ピッキング
1. ダッシュボードから患者を開き、棚番地順のピッキングリストで薬品を集めます。
2. **GS1照合**: 現物のGS1バーコードをスキャンし、薬品・ロット・期限の一致を確認します。未照合が残ると「GS1未照合」フラグが表示されます。
3. **棚不足があったら**: その場で「不足を記録」します。記録した不足は発注ワークベンチの候補に自動で載ります。あとでまとめて、ではなくその場で記録します。
4. **既存ピッキングシステムを使う店舗**: ピッキング支援画面の「指示CSV」を書き出してシステムへ渡し、払出後に「結果取込」で結果CSVを取り込みます。完了行は照合済みに、不足行は不足記録に反映されます。取り込んだ後は、現物とロット・期限が一致しているか必ず確認してください。院内でGS1照合済みの明細は外部結果で上書きされません。

### 3.3 処方監査・薬剤師確認（薬剤師）
1. 患者アラート（アレルギー・副作用歴）、相互作用、重複投薬の警告を確認します。
2. **要修正（赤）**: 解消するまで印刷・UKE出力へ進めません。処方修正または疑義照会で解消します。
3. **薬剤師確認（黄）**: 内容を確認し、問題なければ理由を残して先へ進みます。
4. 疑義照会をした場合は、変更内容・医師回答を記録します（薬歴・調剤録・レセプト摘要に連動します）。

### 3.4 薬歴（SOAP）
1. EMR画面で前回薬歴と処方差分を確認してから服薬指導します。
2. SOAPを入力します。AI下書きは補助であり、**保存前に薬剤師が必ず内容を確認・修正**します。
3. 服薬フォローが必要な患者は、フォロー候補に理由と次回確認日を残します。

### 3.5 印刷・会計・完了
1. 印刷画面で調剤録・薬袋・薬情・お薬手帳シール等を出力します。
2. 未完了の確認事項が残っていると完了できません。表示に従って解消します。
3. 完了操作で在庫が引き落とされ、患者が完了一覧へ移ります。

## 4. 閉局時チェック（毎日 10分）

1. **対応キューを空にする**: 処理中・監査待ちの患者が残っていないか確認します。残す場合は理由をチームで共有します。
2. **日次締め**: ダッシュボードの日次締めで、完了件数・在庫不足・フォロー候補・残タスクを確認し、承認します。
3. **バックアップ**: 「設定 > バックアップ」で暗号化バックアップを書き出し、店舗で定めた外部保存先へ置き、「外部保存を記録」します。自動化している店舗は、状態JSONと失敗通知が出ていないかを確認します（user_manual §6）。
4. **発注**: 発注ワークベンチで当日の不足候補を確認し、発注済みへ更新します。

---

## 5. 月次業務（月初・月末）

1. **レセプト（UKE）**: 月次請求ワークベンチで対象受付を確認し、請求前チェックの指摘をすべて解消してから月次一括UKEを作成します。停止された場合は確認CSVの内容に従って修正します。
2. **受付結果取込・返戻対応**: 審査結果を「結果取込」で反映し、返戻は返戻修正メモと差分CSVで対応します。
3. **監査ログの外部保全**: user_manual §5 の手順で監査ログJSONと保全台帳CSVを出力・保管し、棚卸CSVで不足がないことを確認して責任者承認します。
4. **バックアップ世代確認**: user_manual §6 の手順で3世代以上・暗号化・復旧テスト30日以内を確認します。
5. **患者重複点検（名寄せ）**: 「設定 > バックアップ > 患者重複点検」で「重複候補を確認」を実行し、同姓同名・同生年月日の重複患者がいないか確認します。重複があれば、受付履歴の多い患者を「残す患者」にして統合します（統合は監査ログに残ります）。移行直後の店舗は月1回、安定後は四半期に1回が目安です。
   あわせて「設定 > マスタ更新 > 薬品重複点検」で、使用中薬品の重複（在庫や処方参照が2つのコードに割れている状態）がないかも確認します。マスタ更新の後に実施するのが効果的です。
6. **月次レビュー**: 日次締め月次レビューでKPI（返戻・在庫不足・フォロー・問い合わせ）を確認し、改善アクションを決めます。

## 6. 練習用デモデータの使い方

新人研修や操作練習には、チュートリアルのデモ患者を使います。**実患者で練習しないでください。**

1. 画面右上の「3分デモ」を開き、最後のステップで「**デモ患者で体験を始める**」を押します。
2. 投入される練習データ（すべて「デモ」表記・固定ID付き）:
   - デモ患者 みどり（副作用歴アラート付き: ロキソプロフェンで胃部不快感）
   - 本日の受付1件（処方3剤。3剤目はわざと在庫不足で、不足記録の練習ができます）
   - 前回来局（28日前）の完了受付と薬歴 — 履歴参照・前回Do・処方差分の練習用
   - 在庫ロット（棚番地・JANコード付き）— GS1照合の練習用
3. 受付→ピッキング→薬剤師確認→薬歴→印刷まで、実際の画面で一通り練習できます。
4. **請求への安全装置**: デモ受付はUKEファイル出力・月次一括UKE・外部調剤機器連携の対象になりません。誤って実請求へ混ざることはありません。
5. **練習後の片づけ**: 「3分デモ」最後のステップの「**デモデータを片づける**」で、デモ患者・受付・処方・薬歴・アラート・在庫を一括削除します。削除は監査ログに残ります。デモデータが残っている間はダッシュボード上部に片づけを促すバナーが表示され、そこからも同じ削除を実行できます。日次締めの前に必ず片づけてください。
6. 投入・削除は何度でも安全に繰り返せます。研修のたびに投入→練習→片づけ、を1セットにしてください。

このほか、処方箋受付・ピッキング・薬歴入力の各画面には、保存されない「業務別30秒デモ」もあります（初回に自動表示・画面上のデモボタンで再表示）。

## 7. 困ったとき（トラブル対応）

| 症状 | まずやること | それでも直らないとき |
|---|---|---|
| 画面が固まった | ブラウザの再読み込み（F5）。入力途中のデータは自動保存されている場合があります | 端末再起動。復旧通知が出たら管理者へ |
| データベース復旧の通知が出た | **操作を止める**。他端末での入力も止める | 管理者がバックアップから復旧（user_manual §3.8）。自己判断でブラウザデータを消さない |
| OCRの精度が悪い | 明るい場所で影なく撮り直す。低信頼度項目は原本照合 | 手入力受付へ切り替え |
| UKE出力が止められた | 画面の指摘・確認CSVの内容を1件ずつ解消 | 解消できない指摘は責任者へ。無理に出力しない |
| 在庫が合わない | ロット・期限単位で現物照合し、不足はその場で記録 | 棚卸差異として記録し、原因（返品・破損・記録漏れ）を確認 |
| 端末の故障・紛失 | 予備端末でログインし業務継続（データは各端末のローカルDB + バックアップ） | 最新バックアップから復旧。紛失時は管理者がスタッフ認証情報を再設定 |
| スタッフ退職・パスキー紛失 | 「設定 > 初回セットアップ > 復旧・退職対応」で対象と理由を選んで対応 | 最後の管理者は削除不可。復旧レビューは `npm run staff:access-recovery-review` |

**絶対にやってはいけないこと**
- ブラウザの履歴・キャッシュ・サイトデータの削除（患者データが消えます。バックアップと復旧確認なしでは実施しない）
- 復旧通知が出た状態での入力継続
- デモ以外の実患者データでの練習

## 8. 新スタッフ受け入れチェックリスト

- [ ] スタッフ登録（役割: 事務/薬剤師/管理者）とパスワードまたはパスキー設定
- [ ] 「3分デモ」を一通り視聴
- [ ] デモ患者で受付→ピッキング（GS1照合・不足記録）→薬歴→印刷を1周練習
- [ ] 練習後に「デモデータを片づける」を実行
- [ ] 閉局時チェック（日次締め・バックアップ）を先輩と1回実施
- [ ] トラブル時の連絡先と「やってはいけないこと」を確認
