# pharma-oss 取扱説明書（利用者向けマニュアル）

## 1. はじめに

本システム「pharma-oss」は、次世代の薬局向け電子カルテ（EMR）およびレセプト（UKE）発行システムです。ローカルファーストのデータベース技術を活用し、オフライン環境でも高速なデータアクセスを実現するとともに、直感的なユーザーインターフェースを提供します。

毎日の業務手順（開局前・受付〜完了・閉局時・月次・トラブル対応）を時間割形式でまとめた [現場運用マニュアル](field_operation_manual.md) も併せて参照してください。

## 2. 動作環境

*   **推奨ブラウザ**: Google Chrome（最新版）、Mozilla Firefox（最新版）、Microsoft Edge（最新版）
*   **ネットワーク**: オフライン環境でも基本操作は可能ですが、初回読み込みや一部機能（OCR解析など）ではインターネット接続が必要です。

## 3. 基本的な画面構成と機能

### 3.1. ダッシュボード
システムの入り口となる画面です。現在対応中の患者情報や、直近の処方履歴などを一覧で確認できます。
キーボードショートカット `Cmd + K` (または `Ctrl + K`) で、どこからでも検索・ナビゲーション機能にアクセス可能です。
*   **AI補助予測スコア**: 返戻リスク、在庫欠品、服薬フォロー候補を横断して、優先確認したい候補をスコア、信頼度、根拠付きで表示します。設定が「制限」の場合は高リスク候補だけ、「停止」の場合は候補を表示しません。通常の業務キューはそのまま利用できます。AI補助は候補提示であり、請求・調剤・服薬指導の確定は薬剤師が行ってください。
*   **服薬フォロー候補**: 重点フォロー薬、長期処方、患者アラート、疑義照会をもとに候補を表示します。電話、SMS/メッセージ、来局時、その他の対応方法で記録でき、未完了の場合は次回確認日を残して候補に表示し続けます。

### 3.2. 電子カルテ (EMR) 画面
患者ごとの詳細なカルテ情報を管理します。
*   **SOAPエディタ**: 主観的情報(S)、客観的情報(O)、評価(A)、計画(P)を効率的に入力できる4パネルエディタを備えています。
*   **AI補助 SOAP下書き**: 処方薬、ハイリスク薬、相互作用・患者アラートを根拠に、S/O/A/Pへ反映できる下書き候補を表示します。設定が「制限」の場合は要修正候補だけ、「停止」の場合は下書き候補を表示しません。通常の薬歴入力は継続できます。根拠チップから処方薬、相互作用、患者アラートの該当表示へ戻れます。反映時は薬剤師確認済みのAI採否ログとして記録されます。下書きは候補であり、確定前に必ず内容を確認・修正してください。
*   **処方履歴タイムライン**: 過去の処方履歴や服薬指導の記録を時系列で分かりやすく表示します。
*   **既存ピッキングシステム連携**: ピッキング支援画面の「指示CSV」で、受付ID、明細ID、薬品コード、YJコード、JANコード、必要数量、棚番地、在庫ロット候補を含む指示CSV（UTF-8・形式版付き）を書き出し、フォルダ取込型の既存ピッキングシステムへ渡せます。患者名を含むため店舗内利用限定です。ピッキングシステム側の結果CSV/TSVは「結果取込」から取り込み、完了行はGS1照合と同じ形（照合済み・ロット・使用期限）へ、不足行は不足記録へ反映します。列名は「明細ID/薬品コード/JANコード」「結果(完了・不足)」「数量」「不足数」「ロット」「使用期限」「備考」のゆれを吸収し、明細ID・薬品コード・JAN/GTINの順で処方明細と突合します。院内でGS1照合済みの明細は外部結果で上書きしません（現物照合を優先）。別受付の行・突合できない行は取り込まず、反映前に件数と内容を確認できます。書き出し・取込は監査ログに残ります。在庫の引き落としは行わず、完了操作時の既存フローに従います。

### 3.3. 処方箋OCR読み取り機能
紙の処方箋をスキャン、または画像をアップロードして、テキストデータとして自動入力する機能です。
*   **スマート入力**: ブラウザ上で動作するOCR（光学文字認識）を利用し、患者情報や処方内容を自動で読み取ります。
*   **並べて確認**: 読み取った元画像とテキストデータを左右に並べて表示し、誤認識がないか目視で確認・修正できます。
*   **OCR信頼度と見直し箇所**: OCR本文、患者情報、処方元、処方内容をもとに読み取り信頼度と人手確認ポイントを表示します。低い信頼度や確認ポイントがある場合は、保存前に処方箋原本と照合してください。
*   **電子処方箋受付**: 受付入力画面の「電子処方箋」欄から、電子処方箋ID、または引換番号と被保険者番号を入力してpharma-oss接続モジュールへ取得要求できます。引換番号は現在6桁ですが、最大16桁まで入力できます。「処方内容（控え）」は処方箋そのものではないため、控えだけを見て調剤せず、記載された引換番号から電子処方箋または処方箋情報提供ファイルを取得してください。処方箋情報提供ファイルの場合は紙の処方箋原本を受領し、取得内容と照合した後だけ処方入力へ反映できます。接続先が未設定の場合は取得しません。デモ、受付中の患者と違う結果、入力したID・引換番号・被保険者番号と違う結果、内容不足、電子署名未確認、有効期限切れ、重複投薬・併用禁忌の確認未実施または受付停止結果は処方入力へ反映しません。注意がある場合は薬剤師確認として表示します。
*   **マイナ読取**: カードリーダー連携サービスの接続先が設定されている場合は読取内容を反映します。未設定の環境では、画面上でデモ用データであることを表示します。
*   **オンライン資格確認**: 保険者番号を入力して資格確認を行うと、保険者番号、記号番号、負担割合、有効期間、公費情報を患者マスターへ反映できます。外部接続先が未設定の環境ではデモ結果であることを表示し、結果チップにカーソルを合わせると、どの元項目名を認識したか確認できます。

### 3.4. レセプト(UKE)発行機能
電子レセプト（UKEファイル）の生成を行います。
*   ボタン一つで所定のフォーマットに従ったデータを出力でき、請求業務を大幅に効率化します。
*   **請求前チェック**: 廃止薬、薬価未設定、レセ電コード欠落、保険情報不足に加え、UKEの必須レコード、並び順、件数、合計点数、主要項目の形式を確認し、修正が必要な状態ではUKE出力を止めます。月次一括UKEでは、全項目定義に基づく指摘を受付・患者・レコード項目ごとの一覧CSVとして保存し、作成前に止めた場合も保存ファイル名を確認できます。CSVと操作ログには確認に使った公式PDFの出典URLも残ります。公式サンプルだけで見る種別が通常請求UKEに混ざっていないことも確認し、停止理由とCSV名は操作ログにも残ります。
*   **薬剤師確認とAI補助**: 印刷/UKE出力画面では、処方監査の確認事項をもとにAI補助提案を表示します。各提案には根拠、信頼度、薬剤師確認必須の表示があり、採用・修正・却下を監査ログに残せます。「制限」では要修正候補だけ、「停止」では候補を表示しませんが、通常の処方監査は継続します。AI補助は候補提示であり、請求・調剤・服薬指導の確定は薬剤師が行ってください。
*   **受付結果取込**: ダッシュボードの「結果取込」からCSV/TSV、固定長風テキスト、PDFからコピー・抽出した帳票テキストを選択すると、受付済・返戻対応を請求状態へ反映します。ファイル先頭にタイトル行や医療機関情報がある場合も、受付IDと結果列、または1受付ずつ並ぶラベル項目を自動検出します。
*   **受付済み一括締め**: 月次請求ワークベンチの「受付済締め」から、受付済み請求だけを請求完了へまとめて更新し、受付ごとの請求状態変更と一括締めの監査ログを残せます。
*   **返戻修正メモ**: UKE出力後の請求画面では、請求時点スナップショットと現在値の差分を確認できます。差分CSVを書き出し、患者・保険・処方・点数の修正候補を含む返戻修正メモをコピーできます。CSV書き出しとメモコピーは操作ログに残ります。
*   **外部調剤機器へ送る**: 設定の「外部連携」で施設内コネクタが「設定OK」の場合、帳票画面から確認済みの処方を調剤機器またはPOSへ送れます。送信後は連携IDと時刻を表示し、同じ内容の二重送信を防ぎます。処方を直した場合は「差し替える」、受付を取り消す場合は「連携を取り消す」を使い、理由を記録します。薬剤師確認に要修正が残る処方、接続準備が未完了の環境、送信中の重複操作では送れません。以前の受付画面にあったRak-Che連携フォルダへの自動出力は、仕様未確認のため廃止しています。

### 3.5. 在庫管理
在庫画面では、棚番地、実在庫、入荷データ、発注候補を確認できます。
*   **発注ワークベンチ**: 受付中・調剤中の処方から在庫不足候補を出し、未対応と発注済みに分けて確認できます。発注候補CSVとメモには患者名を含めません。
*   **入庫確認CSV**: 発注済み候補だけを、ロット番号、使用期限、納品数量、入庫日、確認者を記録するCSVとして出力できます。
*   **入庫登録**: 発注済み候補に納品数量、ロット番号、使用期限、入庫日、仕入先を入力すると、ロット在庫と薬品マスター在庫数を更新し、操作ログへ残せます。
*   **日次・月次の確認**: 発注ワークベンチから登録した入庫件数は日次締めに表示され、在庫不足品目数、服薬フォロー候補数、問い合わせ対応の多さとともに月次レビュー、責任者CSV、BI JSONで確認できます。

### 3.6. 操作ログ・監査ログ
設定画面の「操作ログ・監査ログ（監査証跡）」では、誰がいつ操作したか、監査ログが改ざんされていないか、外部保全に必要な最新ハッシュを確認できます。
*   **監査ログ整合性**: 署名済み件数、未署名件数、異常件数、最新ハッシュを確認できます。
*   **監査ログJSON**: 監査提出や保全用に、監査ログ本体をJSON形式で書き出します。JSONには責任者保全欄と外部WORM保存確認欄が含まれます。
*   **保全台帳CSV**: 外部WORM保存先、保持期間、読み取り確認、責任者確認、最新ハッシュ照合を記録するためのCSVを出力します。
*   **S3 WORM事前確認**: 監査ログやバックアップをS3へ保存する前に、`npm run s3-worm:preflight -- --destination s3://<保存先>/ --receipt <事前確認受領書JSON>` を実行すると、AWS認証、バケットVersioning、Object Lock設定、非個人情報の確認用ファイル保存、読み戻しSHA-256照合を確認できます。
*   **監査ログS3 WORM保全ジョブ**: AWS CLIの認証とObject Lock有効バケットを用意したうえで、`npm run audit:s3-worm-retention -- --audit-json <監査ログJSON> --destination s3://<保存先>/` を実行すると、監査ログJSONをS3へ保存し、読み戻しSHA-256照合と最新ハッシュ照合を行い、受領書JSONを出力できます。
*   **監査ログS3 WORM定期保全ジョブ**: `npm run audit:s3-worm-retention:scheduled -- --audit-dir <監査ログJSON置き場> --destination s3://<保存先>/ --failure-notice <失敗通知JSON> --failure-webhook-url <監視Webhook>` をOS側のスケジューラから実行すると、置き場内の最新監査ログJSONを選び、同じ監査ログの二重実行を避けながら状態JSONを残します。失敗時は最新ハッシュ、保存先、エラー、再実行手順を失敗通知JSONへ書き出し、監視WebhookへPOSTできます。
*   **棚卸CSV**: 今月の監査ログJSONと保全台帳CSVの出力状況を確認し、不足がある場合は差し戻し理由と対応内容をCSVで出力します。
*   **責任者承認/差し戻し記録**: 月次棚卸が完了している場合は「責任者承認」、不足がある場合は「差し戻し記録」を押すと、確認者、判定、最新ハッシュ、差し戻し件数を操作ログへ残せます。
*   **日次締め月次レビュー**: 日次締め承認ログから承認日数、平均完了率、残タスク推移、在庫不足、入庫登録、服薬フォロー候補、問い合わせ対応の多さ、それぞれの前月差、直近6か月比較、店舗別KPIベンチマークを確認できます。店舗別KPIでは4指標を日平均で自店、全店、他店比較し、差が大きい場合は在庫不足、服薬フォロー、問い合わせ負荷の改善アクションを表示します。実行記録後は実行後の日次締め3営業日分で効果を確認でき、未実施フォロー、担当者割当、店舗横断フォロー、未完了エスカレーション、延期理由と再期限、患者情報なしのBI JSONも利用できます。
*   **AI補助フィードバック月次レビュー**: AI補助提案の採用、修正、却下、平均信頼度、SOAP下書きのS/O/A/P別件数、提案種別・店舗別の傾向に加え、却下率、高信頼度候補の却下、理由未記入を確認できます。品質ゲートは「継続可」「制限が必要」「停止基準に該当」「評価件数不足」を判定し、現在モードと推奨モードを表示します。管理者は「推奨モードを反映」から変更でき、操作ログに残ります。これは採否ログに基づく安全確認であり、臨床的な正確性の証明ではありません。フィードバックCSVとBI JSONには患者情報を含めません。
*   **AI症例レビュー**: パイロット運用では、AI候補が役に立ったか、誤りがあったか、高信頼度の誤りが出ていないかを薬剤師が確認します。レビューには匿名ケースID、匿名店舗ID、提案種別、信頼度、採否、安全影響だけを使い、個人を特定できる内容や症例本文は入れません。
*   **帳票・実紙検証**: 調剤録、明細書、領収証、薬剤情報、薬袋、手帳シール、水剤ラベル、軟膏ラベルを、実プリンタ、実紙、PDF、ラベル紙で確認します。検証依頼書で、帳票ごとの紙種、想定寸法、確認項目、店舗内だけで扱う情報、サポートへ共有してよい集計値を確認できます。共有成果物には、患者名、薬品名入り原本、スクリーンショットファイル名、ローカルパス、プリンタ名、確認者名を入れません。
*   **移行受入・1日テスト**: 他社ソフトから出した患者、受付、在庫、薬歴データで、導入初日に患者と受付、在庫確認、薬歴参照がつながるか確認します。提出物依頼書で、患者CSV、受付CSV、在庫CSV、薬歴CSV、証跡JSONに必要な列、最低件数、店舗内だけで扱う情報、サポートへ共有してよい集計値を確認できます。個人情報なし診断JSONには件数、参照不整合、初日業務の判定、復旧前プレビュー、1日テスト計画、責任者確認だけが集計値で入り、CSV原文や移行元IDは共有されません。
*   **パイロット店舗KPI**: 複数店舗で4週間以上、返戻、処理時間、閉店前残タスク、欠品、フォロー期限内率、重大障害、問い合わせ件数を匿名店舗IDで確認します。提出依頼書で、週次KPI、更新後レビュー、障害対応、問い合わせ確認、改善アクション、責任者確認に必要な項目と、サポートへ共有してよい集計値を確認できます。患者名、店舗名、スタッフ名、問い合わせ本文、URL、トークン、ローカルパスは共有しません。
*   **パイロット正式運用判定**: 複数店舗の4週間KPIだけでなく、更新運用、移行、帳票実紙検証、AI症例レビュー、オンライン資格確認、電子処方箋の公式運用試験、責任者レビュー、停止ルール、サポート引き継ぎをまとめて確認します。正式運用判定の提出依頼書で、どのレビューJSONをどのコマンドで作り、サポートへ何を共有してよいか確認できます。どれかが未完了の場合は正式運用候補にせず、必要な対応を残します。個人情報なし診断JSONにはパイロットKPIの店舗数、週数、後半悪化、重大障害、証跡品質だけが集計値で入り、サポートへ渡しても店舗名や元資料は共有されません。
*   **リリース運用受入**: 更新準備、更新後レビュー、SLAレビュー、問い合わせ訓練をまとめて、運用を広げてよいか確認します。個人情報なし診断JSONには添付レビュー数、同じ更新としてつながっているか、実問い合わせまたは更新失敗訓練の有無、問い合わせ件数、エラー件数、停止時間、責任者引き継ぎだけが集計値で入り、問い合わせ本文や告知本文は共有されません。
*   **AI補助の表示範囲**: 「設定 > 施設基準 > AI補助運用」で、標準、制限、停止を選べます。標準は根拠付き候補をすべて表示し、制限は要修正候補だけ、停止はAI補助候補を表示しません。新規店舗は制限から開始します。

### 3.7. 初回セットアップ
設定画面の上部にある「初回セットアップ」では、新規店舗がテスト運用を始める前に必要な準備を一覧で確認できます。
*   **薬局基本情報**: 薬局名、保険薬局コード、所在地、電話番号、既定の担当薬剤師が仮設定のままになっていないか確認します。
*   **スタッフ認証**: 管理者、薬剤師、認証情報未登録スタッフを確認し、パスワードまたはパスキー登録へ移動できます。
*   **権限ロール設定**: 管理者は固定のまま、薬剤師・事務に許可する操作を店舗の運用に合わせて調整できます。保存や標準へ戻す操作は操作ログへ残ります。
*   **復旧・退職対応**: 端末移行、スタッフ退職、パスキー紛失時は対象スタッフと理由を選び、変更前バックアップ、管理者が残ること、復旧テストの状況を確認します。必要に応じてパスワード再設定、パスキー解除、退職前チェック記録を行い、操作ログへ残します。最後の認証済み管理者は削除できません。
*   **復旧・退職対応レビュー**: 月次点検や監査前は `npm run staff:access-recovery-review` で、端末移行、スタッフ退職、パスキー紛失の対応を匿名ケースIDと対象ロールだけで確認できます。患者名、スタッフ名、薬局名、監査ログ本文、パスワード、パスキー情報は成果物に入れません。スタッフタブの「月次棚卸CSV」から当月の対象操作件数、月次棚卸の判定、3場面の不足だけを書き出せます。個人情報なし診断JSONにも同じ要約が入るため、サポートへ渡してもスタッフ名やログ本文は共有されません。
*   **導入チェック**: 医薬品マスター更新、移行データ診断、請求テスト、帳票印刷テスト、バックアップ訓練の実施状況を操作ログから確認します。各ステップには必要な対応が表示されるため、CSVを書き出す前でも次の作業を確認できます。
*   **引き継ぎメモ**: 導入準備の判定、次の作業、残対応をコピーし、サポート担当者や責任者へ共有できます。コピーした記録は操作ログにも残ります。
*   **導入E2E証跡**: 請求テストは請求状態変更とUKE出力、帳票印刷テストは印刷実行の監査ログを証跡として判定します。サポート担当者は導入確認用のテストデータとブラウザテストで、対象画面の表示と操作導線を確認できます。
*   **業務別30秒デモ**: 処方箋受付、ピッキング支援、薬歴入力の各画面で、実際の業務を始める前に3手の流れを確認できます。表示する患者名、薬品、受付番号、ロットは練習用の固定データで、患者・受付・在庫・薬歴には保存されません。スタッフごと、業務ごとに初回だけ自動表示し、画面上のデモボタンからいつでも開き直せます。
*   **デモ患者で体験**: 画面右上の「3分デモ」最後のステップから、練習用のデモ患者・受付・処方3剤・在庫ロット（棚番地/JAN付き）に加えて、前回来局（28日前）の完了受付と薬歴、副作用歴アラートを投入できます。受付からピッキング（GS1照合・不足記録）、薬剤師確認、薬歴、印刷までを実際の画面で練習できます。デモ受付はUKEファイル出力・月次一括UKE・外部調剤機器連携の対象にならないため、実請求へ混ざりません。練習後は同じ画面の「デモデータを片づける」でデモ一式を削除でき、投入・削除は監査ログに残ります。デモデータが残っている間はダッシュボード上部にも片づけを促すバナーが表示され、そこからも削除できます。
*   **チェックリストCSV**: 導入準備の判定、証跡、必要な対応をCSVで出力できます。

### 3.8. バックアップ/復旧
設定画面の「バックアップ」では、ローカルDBのバックアップ、復旧前の差分確認、復旧テスト、世代管理を行えます。
*   **バックアップを書き出す**: 患者、受付、処方、薬歴、マスタ、設定、スタッフ、操作ログをJSONとして保存します。パスワード暗号化を有効にして保存してください。
*   **ブラウザ外バックアップ書き出し**: 閉店後などにOSスケジューラから `npm run backup:browser-export -- --user-data-dir <ログイン済みブラウザプロファイル> --download-dir <バックアップ置き場> --password-env YAKUREKI_BACKUP_PASSWORD` を実行すると、設定画面の書き出し導線を自動操作し、暗号化バックアップJSONと実行受領書JSONを出力できます。外部保存連携JSONも同時に出す場合は `--export-transfer-manifest --destination-name <保存先名> --destination-path <保存先パス/URL>` を追加します。
*   **外部保存を確認する**: 書き出したJSONを店舗で定めた保存先に置き、保存先から開けること、上書き・削除されにくい設定であること、確認者を監査ログに記録します。
*   **外部保存連携JSON**: バックアップ書き出し時に、保存対象ファイル名、SHA-256、サイズ、保存先、保持日数、上書き削除不可確認条件をまとめたJSONも出力できます。ブラウザ外の保存ジョブやWORM保管ツールに渡す確認用ファイルとして使います。
*   **外部保存ジョブ**: `npm run backup:external-transfer -- --manifest <外部保存連携JSON>` を実行すると、マウント済み保存先へバックアップをコピーし、読み戻してSHA-256を再確認し、受領書JSONを出力できます。S3などのクラウドURLは専用コネクタが必要なため、このジョブでは成功扱いにしません。
*   **定期実行用の外部保存ジョブ**: `npm run backup:external-transfer:scheduled -- --manifest-dir <連携JSON置き場> --failure-notice <失敗通知JSON> --failure-webhook-url <監視Webhook>` をcron、launchd、タスクスケジューラなどから実行すると、置き場内の最新連携JSONを選び、保存先がローカル/マウント先かS3 WORMかを自動で振り分け、同じバックアップの二重実行を避けながら状態JSONを残せます。失敗時は対象連携JSON、保存先、エラー、再実行手順を失敗通知JSONへ書き出し、監視WebhookへPOSTできます。S3へ送る場合は、同じコマンドに `--aws-bin`、`--profile`、`--region` などを追加できます。WebhookにBearerトークンが必要な場合は `--failure-webhook-bearer-env <環境変数名>` を使います。
*   **定期ジョブ点検・通知訓練**: `npm run ops:schedule-drill -- --scheduler-name <ジョブ名> --backup-state <バックアップ状態JSON> --audit-state <監査ログ状態JSON> --browser-profile-path <ログイン済みブラウザプロファイル> --s3-preflight-receipt <事前確認受領書JSON> --scheduler-evidence <スケジューラ登録証跡> --receipt <点検受領書JSON> --webhook-url <監視Webhook>` を実行すると、定期ジョブの状態JSON、受領書、ブラウザプロファイル、スケジューラ登録証跡を確認し、患者情報なしの到達訓練イベントを監視WebhookへPOSTできます。
*   **S3 WORM事前確認**: S3保存を初めて使う店舗では、実データを送る前に `npm run s3-worm:preflight -- --destination s3://<保存先>/ --receipt <事前確認受領書JSON>` を実行し、AWS認証、権限、Object Lock、読み戻しを非個人情報の確認用ファイルで点検します。
*   **S3 WORM外部保存ジョブ**: AWS CLIの認証とObject Lock有効バケットを用意したうえで、`npm run backup:s3-worm-transfer -- --manifest <外部保存連携JSON>` を実行すると、S3へObject Lock付きで保存し、読み戻しSHA-256照合と保持設定確認を行い、受領書JSONを出力できます。
*   **受領書JSONを記録**: 外部保存ジョブが出力した受領書JSONを選択すると、ファイル名、保存先、SHA-256、読み戻し結果、上書き削除不可確認を外部保存確認として監査ログに残せます。
*   **患者重複点検（名寄せ）**: 「重複候補を確認」で、氏名またはカナと生年月日が一致する患者を全件から洗い出します。グループごとに「残す患者」（受付件数が多い患者を推奨表示）を選び、統合確認で受付・アラートの移動件数と保険情報等の差分を確認してから統合を実行します。統合元患者は削除され、実行は監査ログに残ります。受付画面の「同姓同名の統合確認」と同じ統合処理を使い、失敗時は適用済みの操作を自動で取り消します。チュートリアルのデモ患者は対象外です。
*   **移行CSVをプレビュー**: 既存薬局ソフトから出力した患者CSV/TSV、受付CSV/TSV、在庫CSV/TSV、薬歴CSV/TSVを、患者ID、氏名、生年月日、保険情報、受付ID、来局日、薬品コード、ロット、使用期限、在庫数、S/O/A/Pなどの列名ゆれ込みで読み取り、移行用バックアップJSONとして復旧前プレビューに回せます。
*   **復旧前プレビュー**: 復旧前に差分と整合性を確認し、導入移行診断でID欠落、同一ID重複、文字化け疑い、必須領域不足を確認できます。
*   **閉店時バックアップ予定**: 閉店時刻に合わせて予定時刻を設定すると、今日の暗号化バックアップと外部保存確認が未完了の場合にダッシュボードと日次締めで要対応として表示します。
*   **バックアップ世代管理**: 直近7日で3世代以上のバックアップがあるか、暗号化されているか、30日以内に復旧テストがあるかを確認できます。
*   **世代管理CSV**: バックアップ世代数、暗号化有無、最新復旧テスト、外部保存確認、必要な対応をCSVで出力します。

### 3.9. 医薬品マスター更新
設定画面の「マスタ更新」では、支払基金などから取得した医薬品マスターCSVまたはZIPを取り込みます。
*   **差分CSV**: 更新後に、新規・更新・廃止の件数と対象薬品を確認するCSVが自動で書き出されます。
*   **ロールバックJSON**: 更新前の既存行と新規追加行の削除対象を含むJSONが自動で書き出されます。店舗で定めた保管先に、更新元CSVと一緒に保存してください。
*   **ロールバック**: 更新内容に誤りがあった場合は、同じ画面でロールバックJSONを選択し、更新前の薬価・YJ・廃止状態へ戻します。
*   **列定義チェック**: ヘッダー付きCSVは列名を確認し、支払基金標準列のCSVは令和8年基本マスターファイルレイアウトの医薬品マスター42項目と照合して取り込みます。必須列や標準42項目が確認できない場合は更新を止めます。監査ログとロールバックJSONには、照合に使った支払基金ファイルレイアウトPDFの版、PDF URL、先頭/末尾項目の確認結果も残ります。
*   **仕様PDF本文照合**: 「公式PDFを取得して照合」で支払基金ファイルレイアウトPDFを取得し、検索可能PDFの本文を抽出して現在の42項目定義との違いを確認できます。画像化PDFなどで本文を読めない場合は読取不可として止まるため、OCRまたはPDF変換ツールで抽出した本文を貼り付けて、項番、項目名、モード、桁数、バイト数を照合してください。外部OCRツールから連携する場合は、医薬品マスター仕様PDF本文を `/api/drug-master/official-spec-pdf` へPOSTすると同じ照合結果を取得できます。
*   **ZIP取り込み**: 全件ファイルなどのZIPは画面内でCSVを展開します。ZIP内CSV名、ZIP内ファイル数、CSV候補数は監査ログとロールバックJSONの証跡に残ります。
*   **更新元URL**: 入力すると、監査ログとロールバックJSONに更新元URL、公式URL確認結果、ファイルサイズ、SHA-256が記録されます。
*   **更新候補選択**: 「公式ページを取得」で支払基金の医薬品マスター掲載ページを読み込み、全件、改定分、改定内容などの候補を更新元URLへ反映できます。CSV/ZIP候補は「更新元URLから取得して更新」で、そのまま取得して取り込めます。HTMLを貼り付けて候補抽出することもできます。
*   **PMDAラベルキュー確認**: PMDA公式添付文書由来の相互作用・禁忌データを更新した後は、`npm run drug-label:queue-review` で未処理、通信失敗、PMDA候補なし、要確認、古い形式のデータ混入を確認します。PMDA候補なしは `npm run drug-label:no-candidate-review` で人確認と責任者確認を残してから閉じます。保留が残る場合は、薬歴画面の警告データを出荷候補として扱いません。
*   **薬品重複点検（マスタ統合）**: 「重複候補を確認」で、YJコードまたは薬品名が一致する薬品のうち、店舗で使用中（在庫・処方参照・棚番地あり）のものを洗い出します。一般名処方マスタ【般】行・デモ薬品・店舗未使用のマスタ由来重複は対象外です。「残す薬品」を選んで統合すると、在庫ロットと処方参照を残す薬品へ付け替え、在庫数を合算し、統合元を削除します（実行は監査ログに残ります）。YJコードが異なる薬品は別薬品の可能性があるため統合できません。統合元コードの薬情テンプレ・指導文は付け替えないため、必要な場合は統合先コードで作成・承認し直してください。過去に出力したUKE・請求時点スナップショットは変更されません。

### 3.10. 公式仕様点検
設定画面の「公式仕様点検」では、公開資料に対する実装状況と残作業を確認できます。これは保健所などの外部監査ではなく、薬局内で行う自己点検です。
*   **UKE仕様PDF 全項目確認**: 「公式PDFを取得して確認」で支払基金の調剤UKE記録条件仕様PDFを取得し、対象レコード数、抽出項目数、定義済み項目数、残項目数を確認できます。PDFの文字を正しく取り出せない場合は読取不可として止めます。手元でPDFから取り出した文字を貼り付けて確認することもできます。実装確認メモを入れると、項目一覧、実装計画、追加候補、反映案、残対応をまとめた実装パックを書き出せます。
*   **公式提出形式の骨格確認**: 支払基金の令和8年6月版調剤記録条件仕様 `iryokikan_in_07.pdf` に基づき、先頭YK、末尾GO、必須レコード、標準レコード種別を点検できます。SN/JD/MF/STは条件に応じた任意レコードとして扱い、KHなど仕様一覧にないレコードがある場合は修正が必要と判定します。
*   **確認結果CSV**: 完了・未完了の判定、確認を止めている理由、対象レコード、次の対応をCSVへ書き出せます。確認実行とCSV書き出しは操作ログに残ります。

### 3.11. 薬情テンプレ承認
設定画面の「薬情テンプレ」では、患者へ印刷する「副作用・相談目安」と「使用上の注意」を薬品ごとに管理します。
*   **文章案の扱い**: 「副作用/注意案」と「注意案CSV」は編集開始用の下書きです。添付文書や患者向医薬品ガイドなどの公式資料と照合し、対象薬に合わない表現を修正してください。文章案やCSV取込だけで承認済みにはなりません。
*   **承認前の確認**: 2つの本文、参照元区分、参照元版日、参照元URLまたはハッシュ・管理番号を入力します。画面に表示される不足項目がなくなったことを確認してから、薬剤師が「承認して保存」を押します。
*   **参照元の記録**: 「PMDA検索URL」は公式資料を探すための検索先です。検索結果から確認した資料の版日を入力し、URLまたは店舗で追跡できる管理番号を残してください。参照元版日に未来日は指定できません。検索結果や他サイトの文章を無確認で転載する用途ではありません。参照元区分をPMDA添付文書またはPMDA患者向医薬品ガイドにする場合、参照元URLは `https://pmda.go.jp` 配下のURLにしてください。「薬のしおり」等の転載元URLは承認根拠として扱わず、承認前の不足として表示されます。
*   **状態別の確認**: 一覧上部の「下書き」「承認済み」「要再確認」「廃止」で絞り込めます。「承認準備OK」は本文と参照元が揃った状態を示すだけで、薬剤師の承認操作が終わった状態ではありません。検索結果が80件を超える場合は先頭80件だけを表示するため、薬品名または薬品コードを併用してください。
*   **CSV取込結果**: CSV下書き取込後は、取込件数、承認準備OK件数、不足・警告件数が画面に残ります。CSV書出では参照元区分を「PMDA 添付文書」「薬局作成」などの日本語ラベルで出力します。取込時は `pmda_insert` などの内部値に加え、「PMDA 添付文書」「PMDA 患者向医薬品ガイド」「薬局作成」「許諾済み資料」「その他」などの日本語ラベルも使えます。不足ありのテンプレは一覧の承認準備フィルタで絞り込み、本文や参照元を補ってから承認してください。
*   **版の保護**: 承認済み、要再確認、廃止の本文や参照元を編集すると、新版の下書きに切り替わります。下書き保存または承認保存では新しいテンプレIDを作り、元の版の本文を上書きしません。本文を変更した状態では、元の版を「要再確認」や「廃止」へ変更できません。
*   **印刷時の動作**: 承認条件を満たす最新版だけが薬情へ印刷されます。未作成、下書き、要再確認、承認不備の薬剤は個別本文を出さず、安全な定型文とPMDA公式検索リンクへ切り替わります。

## 4. 基本操作の流れ

1.  **患者の登録・検索**: ダッシュボードまたは検索機能から、該当する患者を選択するか、新規登録を行います。
2.  **処方箋の入力**:
    *   **手入力の場合**: EMR画面から処方内容を入力します。
    *   **OCR入力の場合**: 処方箋画像をアップロードし、自動入力された内容を確認・修正します。
3.  **カルテの入力**: EMR画面のSOAPエディタを使用し、服薬指導等の記録を記入して保存します。
4.  **レセプトデータの出力**: 月末・月初に、レセプト発行機能から対象期間を指定してUKEファイルを出力します。

## 5. 監査ログの外部保全

月次締め、責任者レビュー、監査対応の前後では、以下の手順で監査ログを外部保全します。

1.  設定画面で「操作ログ・監査ログ（監査証跡）」を開きます。
2.  「監査ログ整合性」が「正常」であることを確認します。未署名や異常が表示される場合は、出力前に内容を確認してください。
3.  「監査ログJSON」を押して、監査ログ本体を書き出します。
4.  「保全台帳CSV」を押して、外部保全の確認用台帳を書き出します。
5.  監査ログJSONを、店舗で定めた外部保管先に保存します。WORM、オブジェクトロック、読み取り専用媒体など、保存後に上書き・削除されにくい設定を利用します。
6.  S3 Object Lockバケットを使う場合は、初回と設定変更時に `npm run s3-worm:preflight -- --destination s3://<保存先>/ --receipt <事前確認受領書JSON>` を実行し、受領書JSONを保管します。
7.  S3 Object Lockバケットへ保存する場合は、`npm run audit:s3-worm-retention -- --audit-json <監査ログJSON> --destination s3://<保存先>/ --expected-latest-hash <画面の最新ハッシュ>` を実行し、受領書JSONを保管します。月次や閉店後の運用へ載せる場合は、`npm run audit:s3-worm-retention:scheduled -- --audit-dir <監査ログJSON置き場> --destination s3://<保存先>/ --failure-notice <失敗通知JSON> --failure-webhook-url <監視Webhook>` をOS側のスケジューラに登録します。
8.  保存先から監査ログJSONを開けることを確認し、JSON内の最新ハッシュと保全台帳CSVの最新ハッシュが一致することを確認します。S3 WORM保全ジョブを使った場合は、受領書JSONの `latestHashMatched` が `true` であることも確認します。
9.  保全台帳CSVに保存先名、保存先パスまたはURL、保持期間、確認者、確認日時、備考を記録します。
10.  月次締め時は「棚卸CSV」を出力し、差し戻し理由が「なし」になっていることを確認します。差し戻し理由が表示される場合は、CSVの対応内容に従って監査ログJSONまたは保全台帳CSVを再出力してください。
11.  定期保全に載せている場合は、`npm run ops:schedule-drill -- --scheduler-name <ジョブ名> --audit-state <監査ログ状態JSON> --receipt <点検受領書JSON> --webhook-url <監視Webhook> --require-audit-state` を実行し、状態JSONとWebhook到達訓練の受領書を保管します。
12.  棚卸結果に問題がなければ「責任者承認」を押します。差し戻し理由が残る場合は「差し戻し記録」を押し、対応後に再度CSVと責任者承認を行います。

この外部保全手順は、保健所や支払基金などの外部機関による監査そのものを代替するものではありません。薬局内で、監査証跡を後から確認できる状態に保つための自己点検・保全手順です。

## 6. バックアップの世代確認

閉店時または月次締め前に、以下の手順でバックアップの世代を確認します。

1.  設定画面で「バックアップ」を開きます。
2.  「閉店時バックアップ予定」の予定時刻が店舗の閉店後運用に合っていることを確認します。
3.  「バックアップ世代管理」が「世代OK」であることを確認します。
4.  「保存世代」が3世代以上、「暗号化」が保存世代数と同じであることを確認します。
5.  「復旧テスト」が30日以内であることを確認します。
6.  「最新外部保存確認」が記録済みであることを確認します。未記録の場合は、バックアップJSONを店舗で定めた保存先に置き、「外部保存を記録」を押してください。
7.  毎日の自動書き出しに載せる場合は、ログイン済みブラウザプロファイルとバックアップ用パスワード環境変数を用意し、`npm run backup:browser-export -- --user-data-dir <ログイン済みブラウザプロファイル> --download-dir <バックアップ置き場> --password-env YAKUREKI_BACKUP_PASSWORD --export-transfer-manifest --destination-name <保存先名> --destination-path <保存先パス/URL>` をOS側のスケジューラに登録します。
8.  外部保存ツールへ渡す場合は「外部保存連携JSONも出力する」を有効にし、保存先名、保存先パス/URL、保持日数を確認してからバックアップを書き出します。
9.  保存先がNAS、外付けドライブ、読み取り専用ボリュームなどローカルにマウントされている場合は、`npm run backup:external-transfer -- --manifest <外部保存連携JSON>` で保存ジョブを実行し、受領書JSONを保管します。毎日の運用に載せる場合は、`npm run backup:external-transfer:scheduled -- --manifest-dir <連携JSON置き場> --failure-notice <失敗通知JSON> --failure-webhook-url <監視Webhook>` をOS側のスケジューラに登録します。連携JSONの保存先が `s3://` の場合、この定期実行ジョブはS3 WORM外部保存ジョブへ自動で振り分けます。
10.  保存先がS3 Object Lockバケットの場合は、初回と設定変更時に `npm run s3-worm:preflight -- --destination s3://<保存先>/ --receipt <事前確認受領書JSON>` を実行し、AWS CLIの認証情報、権限、バケットの保持設定を確認します。
11.  事前確認が通ったら、`npm run backup:s3-worm-transfer -- --manifest <外部保存連携JSON>` を実行し、受領書JSONを保管します。
12.  定期実行に載せている場合は、`npm run ops:schedule-drill -- --scheduler-name <ジョブ名> --backup-state <バックアップ状態JSON> --browser-profile-path <ログイン済みブラウザプロファイル> --receipt <点検受領書JSON> --webhook-url <監視Webhook> --require-backup-state --require-browser-profile` を実行し、状態JSON、プロファイル、Webhook到達訓練の受領書を保管します。
13.  設定画面で受領書JSONを選択し、「受領書を監査ログへ記録」を押します。
14.  「世代管理CSV」を出力し、店舗で定めた保存先に保管します。

「要確認」または「世代不足」と表示される場合は、バックアップを書き出し、外部保存確認と復旧テストを記録してください。

## 7. よくある質問・トラブルシューティング

*   **Q: オフライン時にデータは保存されますか？**
    *   A: はい。データはブラウザ内のデータベース（IndexedDB）に保存されるため、一時的にネットワークが切断されても入力や保存が可能です。
*   **Q: 保全台帳CSVだけ保存すれば監査ログの保全になりますか？**
    *   A: いいえ。保全台帳CSVは確認記録です。監査ログ本体である監査ログJSONも、必ず外部保管先に保存してください。
*   **Q: バックアップは1つあれば十分ですか？**
    *   A: いいえ。直近7日で3世代以上を目安に保存してください。誤操作や破損に備えるため、複数世代を残すことが重要です。
*   **Q: 医薬品マスター更新を元に戻せますか？**
    *   A: はい。更新時に出力されたロールバックJSONを「設定 > マスタ更新」で選択し、「ロールバックを実行」を押してください。新規追加された行は削除され、既存行は更新前の内容へ戻ります。
*   **Q: OCRの読み取り精度が低い場合は？**
    *   A: 画像の解像度が低い、または影ができている可能性があります。明るい場所で、処方箋の文字がはっきりと見えるようにスキャン（撮影）し直してください。
*   **Q: 画面が動かなくなった場合は？**
    *   A: お使いのブラウザの「更新」ボタン（または F5 キー）を押して、ページを再読み込みしてください。入力途中のデータは自動的に保存されている場合があります。
*   **Q: 練習で作ったデモ患者やデモ在庫はどう消せますか？**
    *   A: 画面右上の「3分デモ」を開き、最後のステップの「デモデータを片づける」を押してください。デモ患者・受付・処方・薬歴・アラート・在庫が一括削除され、削除は監査ログに残ります。なお、デモ受付はUKE出力・月次一括UKE・外部調剤機器連携の対象にならないため、片づける前でも実請求へ混ざることはありません。
